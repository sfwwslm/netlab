# netlab 设计文档 V1（精简版目录结构）

> **文档定位声明（必须放在最前）**
>
> 本文档定义 netlab 的 **顶层设计目标、架构原则与长期约束**。
> 本文档 **不描述具体实现细节**，也 **不替代各 crate 的 README**。
>
> * engine 的设计与开发规范详见 `crates/engine/README.md`
> * cli 的定位说明详见 `crates/cli/README.md`
> * GUI / Tauri 为可插拔壳层，不是核心

---

## 1. 项目定位（What & Why）

### 1.1 netlab 是什么

* 一个网络行为实验与验证平台
* 用于构造、观察、分析网络与协议行为

### 1.2 netlab 不是什么

* 不是单一 HTTP 压测工具
* 不是 GUI 优先项目
* 不是通用网络库

---

## 2. 设计目标与非目标

### 2.1 设计目标（Goals）

* 核心逻辑独立于 UI
* 网络行为可复现、可验证
* 支持 CLI / GUI / Headless 多入口

### 2.2 非目标（Non-Goals）

* 覆盖所有网络协议
* 面向非技术用户
* 提供“傻瓜式”一键压测

---

## 3. 顶层架构概览（不含细节）

### 3.1 逻辑分层

```text
frontend (UI)
    ↓
tauri (adapter)
    ↓
engine (core)
    ↓
hyper / system
```

> 各层仅通过明确边界交互，不得越界。

---

## 4. 模块角色说明（定位级）

### 4.1 engine（核心）

* 唯一承载网络行为与调度逻辑的模块
* 可独立运行
* 不依赖任何 UI 技术

> 详细设计见 `crates/engine/README.md`

---

### 4.2 cli（次要入口）

* engine 的命令行入口
* 当前优先级最低
* 不承担核心逻辑

> 定位说明见 `crates/cli/README.md`

---

### 4.3 GUI / Tauri（可插拔壳）

* 用于参数配置与可视化
* 可完全移除而不影响 engine

---

## 5. 架构红线（Hard Rules）

> **违反以下任意一条，视为架构错误**

* engine 禁止依赖 tauri / GUI
* engine 禁止出现 UI / JSON / 前端事件语义
* GUI / CLI 不得实现并发、调度或指标计算
* 所有入口必须通过 engine API 执行核心逻辑

---

## 6. 演进策略与稳定性原则

### 6.1 稳定性优先级

1. 架构原则（本文档）
2. engine 行为语义
3. CLI / GUI 表现层

> 早期阶段允许对 engine API 做必要重构，以提升一致性与可解释性；
> 进入稳定期后再强调兼容与语义不变。

---

### 6.2 允许的演进方向

* 新协议支持
* 新网络行为模型
* 新调度策略
* 新观测指标

---

### 6.3 不鼓励的演进方向

* 为单一功能牺牲架构边界
* 在壳层引入“临时逻辑”
* 将 engine 设计绑定到 UI 形态

---

## 7. 文档与工程的关系

* 本文档 **更新频率极低**
* README / 文档位于各 crate 内
* 代码可以迭代，但边界不应轻易改变
* 若核心边界或 engine API 发生调整，必须同步更新文档

---

## 8. 阶段性说明（V1 范围）

* 当前阶段以 engine 为核心
* CLI 提供可用入口（TUI），但不作为核心重点
* GUI 在 engine 稳定后再系统推进

---

## 9. 结束语

> **netlab 的长期价值不在于“功能数量”，
> 而在于“对网络行为的理解深度”。**
