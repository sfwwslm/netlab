# CONTRIBUTING.md

## netlab 贡献指南（V1）

本文档定义 **netlab 项目的贡献规则与工程约束**。
所有代码贡献（包括未来的你自己）都必须遵守本文件。

> **如果某个改动与本文档冲突，
> 即使功能“看起来能用”，也会被拒绝。**

---

## 1. 项目当前阶段说明（非常重要）

netlab 当前处于 **核心引擎（engine）优先阶段**：

* engine：**唯一高优先级模块**
* CLI：可用入口，但不冻结接口
* GUI / Tauri：**以适配 engine 为优先，不做复杂功能堆叠**

因此：

* **允许**：engine 相关改动（含必要 API 重构）
* **谨慎**：CLI 相关改动
* **拒绝**：在壳层引入核心逻辑或提前堆叠 GUI 复杂度

---

## 2. 核心架构红线（Hard Rules）

以下规则 **不可破坏**，违反任意一条，PR 将被直接拒绝。

### 2.1 engine 的绝对独立性

* engine **禁止依赖**：

  * tauri
  * GUI / 前端
  * WebView
* engine **禁止出现**：

  * UI 状态
  * 前端事件名
  * JSON 序列化为 UI 服务的结构

> engine 只关心：**网络、时间、调度、指标**

---

### 2.2 壳层不得侵入核心逻辑

* CLI / GUI：

  * ❌ 不得实现并发控制
  * ❌ 不得实现调度策略
  * ❌ 不得计算核心指标
* 所有核心行为 **必须通过 engine API**

---

### 2.3 不允许“临时方便写法”

以下行为视为架构破坏：

* “先写在 GUI 里，之后再抽”
* “这里写一点逻辑比较省事”
* “只是临时用一下，不影响整体”

👉 **netlab 明确拒绝“先能跑再说”的演进方式**

---

## 3. 模块职责边界

### 3.1 engine

**允许：**

* 网络连接模型
* 协议行为实验
* 调度器实现
* 指标采集与统计

**禁止：**

* UI 语义
* 命令行参数解析
* 输出格式定制（JSON / 表格）

---

### 3.2 cli

**定位：**

* engine 的命令行入口
* 用于 headless / CI 场景

**约束：**

* CLI 不得绕过 engine API
* CLI 不得成为“第二套实现”
* CLI 的设计不得反向约束 engine

---

### 3.3 GUI / Tauri

**当前阶段：**

* 非开发重点
* 仅允许基础 glue code

任何 GUI 功能设计，必须以 **engine 能独立运行** 为前提。

---

## 4. 代码风格与质量要求

### 4.1 通用要求

* Rust 代码必须：

  * 可读
  * 可解释
  * 可维护
* 禁止：

  * 魔法常量
  * 隐式行为
  * 无注释的复杂逻辑

---

### 4.2 engine 的额外要求

* 时间测量统一使用 `Instant`
* 指标优先使用分布（p50 / p90 / p99）
* 不以“平均值”为核心判断依据
* 网络行为应尽量可复现

---

## 5. 提交（Commit）规范

### 5.1 提交粒度

* 一个 commit 只做一件事
* 不混合：

  * 重构
  * 功能
  * 格式调整

---

### 5.2 Commit Message 建议格式

```text
engine: add http1 keep-alive scenario
engine: refactor scheduler lifecycle
cli: initial placeholder structure
docs: clarify architecture hard rules
```

---

## 6. Pull Request 审核原则

以下情况将 **直接拒绝 PR**：

* 破坏 engine / 壳层边界
* 在壳层实现核心逻辑
* 为短期功能牺牲长期架构
* “以后再改”的设计

PR 审核更关注：

* 架构一致性
* 可维护性
* 是否增加技术债

---

## 7. 文档同步要求

以下改动 **必须同步更新文档**：

* 架构原则变更 → `ARCHITECTURE_V*.md`
* engine 设计变更 → `crates/engine/README.md`
* 贡献规则变更 → `CONTRIBUTING.md`

---

## 8. 关于“未来的你”

> 本文件不仅约束他人，
> **同样约束未来的维护者（包括作者本人）。**

如果你发现自己想绕过某条规则：

* 请先修改文档
* 再修改代码
* 而不是反过来

---

## 9. 最终说明

netlab 的目标不是：

* 功能多
* UI 漂亮
* 跑得快就行

而是：

> **对网络行为有清晰、可验证的理解**

所有贡献，必须服务于这个目标。
